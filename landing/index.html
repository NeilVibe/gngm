기술 개요 문서 AI 코딩 에이전트 지식 스택 GNGM + SDP + NLF

GNGM: AI 코딩 에이전트를 위한 4계층 지식 스택

knowledge graph · symbolic code graph · canonical wiki · cross-session entity graph. 서로 다른 자료구조가 서로 다른 유형의 지식 질문에 답하며, SDP/NLF 공학 규율이 이 지식을 정직하게 활용하도록 보장합니다.

요약

AI 코딩 에이전트는 기본적으로 stateless하게 동작합니다 — 세션이 끝나면 작업 문맥이 사라지고, 동일한 지식을 매 세션마다 다시 유도해야 합니다. GNGM은 이 제약을 4개의 이질적 저장소로 분산 해결합니다: LLM이 추출한 사실 그래프(Graphiti), 증상별 경험과 정제된 위키(NeuralTree), AST 기반 코드 구조 그래프(Graphify), cross-session 엔티티 그래프(MemoryMCP). 각 계층은 grep이나 semantic search로는 답할 수 없는 고유한 질문 유형을 담당합니다.

§1 · 문제 정의

기존 검색 도구로는 답할 수 없는 질문들

AI 코딩 에이전트가 답해야 할 질문은 세 가지 층위로 나뉘며, 각 층위마다 적합한 자료구조가 다릅니다. 단일 도구로 모든 층위를 커버하려 하면 반드시 한쪽에서 실패합니다.

AI 코딩 에이전트의 근본적 제약은 두 가지입니다. 첫째, 실행 모델이 stateless하다는 점 — 세션 종료 시 context window의 내용이 모두 소실되므로, 이전 세션에서 확립된 의사결정 맥락, 버그 수정 이력, 사용자 선호 같은 정보가 다음 세션으로 이어지지 않습니다. 둘째, context window가 유한하다는 점 — 프로젝트 전체 코드와 문서를 매번 로드할 수 없으므로 선별적 검색이 필수입니다.

이를 해결하기 위해 보통 RAG 패턴이 쓰이지만, 단일 retrieval 방식(대개 벡터 기반 의미 검색)으로는 다음과 같은 질문들에 제대로 답할 수 없습니다.

질문 유형 예시 기존 도구가 실패하는 이유
구조 (structural) "이 함수를 호출하는 모든 함수는?" grep은 문자열 일치만 함. 의미 검색은 구조를 모름. AST 그래프가 필요함.
관계 (relational) "A 컴포넌트 변경이 B에 미치는 영향은?" 벡터 유사도는 "비슷한 주제"만 찾음. 인과 관계는 그래프 순회가 필요함.
시간 (temporal) "왜 이 방식으로 결정되었고 언제 바뀌었나?" 현재 코드 상태만으로는 결정 이력을 복원 불가. 시간 레이어가 필요함.
경험 (episodic) "이 증상을 전에 고친 적 있나?" 코드에는 버그 수정 과정이 남지 않음. 증상-원인-수정 단위의 경험 기록이 필요함.
선호 (preferential) "사용자가 이 상황에서 원하는 방식은?" 프로젝트 파일에 존재하지 않음. 별도 persistent store가 필요함.

GNGM의 설계 전제는 이 다섯 질문 유형이 서로 다른 자료구조를 요구한다는 것입니다. 하나의 만능 저장소 대신 네 개의 특화 저장소를 병렬로 쿼리하고, 결과를 종합해 답을 구성합니다.

§2 · 4계층 구조

각 계층의 기술적 구성과 이론적 근거

네 계층은 독립된 도구이지만, 하나의 trigger(GNGM)로 동시에 호출되어 결과를 병합합니다. 각 계층은 고유한 질문 유형에 대해 권위 있는 답을 제공하며, 담당 밖의 질문에는 답하지 않습니다.

G₁ Graphiti temporal knowledge graph

정의

Zep Inc.에서 오픈소스로 공개한 temporal knowledge graph 프레임워크. LLM이 자연어 텍스트(여기선 작업 로그, 의사결정 메모 등)에서 엔티티와 관계를 추출해 시간축이 있는 그래프에 저장합니다. 이 구축 환경에서는 Qwen 3.5 9B 모델을 Ollama로 로컬 실행하여 추출하고, FalkorDB(Redis Graph fork)에 저장합니다.

이론적 근거

지식은 사실(facts)과 시간(temporal validity)의 결합입니다. "X는 Y를 호출한다"는 명제는 2024년 3월엔 참이었지만 리팩토링 이후 거짓일 수 있습니다. 전통적인 RDF/OWL 기반 그래프는 사실만 저장하고 시간 차원이 없어, 충돌하는 사실을 화해시키지 못합니다. Graphiti는 각 엣지에 valid_at, invalid_at을 부여해 temporal resolution — 즉 "현재 시점에 참인 사실"을 자동으로 선별합니다. 충돌 시 최신이 우선하며, 과거 사실은 supersede되어 보관됩니다.

검색 메커니즘

단일 방식이 아닌 hybrid search: ① Model2Vec 기반 벡터 유사도 (256차원 다국어 임베딩) ② BM25 lexical score ③ 그래프 순회 보너스 (쿼리 엔티티의 인접 노드 가중치 상승)를 합산합니다. 의미 유사도만으로는 놓치는 구조적 관계와, 구조만으로는 놓치는 의역을 동시에 포착합니다.

담당 질문

  • "어떤 엔티티들이 X와 연결되는가" (blast radius 분석)
  • "왜 이 방식으로 결정되었는가" (의사결정 이력)
  • "최근 Y에 대해 무엇이 바뀌었는가" (temporal query)
LIMITS LLM 추출에 의존하므로 입력 텍스트가 부실하면 엔티티 누락 발생. "정확히 이 함수를 호출하는 함수 목록" 같은 결정적(deterministic) 구조 쿼리는 약함 — Graphify가 담당.
N NeuralTree lesson store + canonical wiki pipeline

정의

증상 기반 lesson 저장소와, 그 lesson들을 자동으로 정제된 위키 페이지로 증류하는 파이프라인. 각 lesson은 symptom · root_cause · fix · chain · key_file이라는 고정 스키마로 기록되어, 증상 문자열로 과거 수정 이력을 검색할 수 있습니다.

이론적 근거 — Karpathy의 WikiLLM 패턴

Andrej Karpathy가 제시한 통찰: "좋은 학습 데이터의 모습은 Wikipedia다" — 즉 산발적 경험 로그가 아니라, 정제되고 서로 링크된 canonical 문서여야 재사용 가능합니다. NeuralTree는 이를 두 단계로 구현합니다. ① 세션마다 atomic lesson을 축적 ② 도메인당 3개 이상 쌓이면 compile()이 LLM 합성을 통해 .neuraltree/wiki/<domain>.md로 증류. 증류된 위키는 의미 검색 인덱스(Viking)에 재투입되어 다음 세션이 재활용합니다.

왜 자동 증류인가

수동 문서화는 소프트웨어 공학에서 가장 오래된 실패 패턴 중 하나입니다 — 개발자가 꾸준히 하지 않습니다. 원자 단위 lesson 기록은 진입 장벽이 낮아 유지되고, 증류는 LLM이 담당하므로 자동으로 scale됩니다. 이는 수동 문서화 대비 compliance 비용을 크게 낮추면서도 지식의 재활용 가능성을 유지합니다.

담당 질문

  • "이 증상을 전에 해결한 기록이 있는가"
  • "이 도메인의 표준적인 수정 패턴은 무엇인가"
  • "반복되는 pitfall은 무엇인가"
LIMITS 기록되지 않은 경험은 찾을 수 없음 — post-fix 단계에서 lesson_add 호출이 반드시 일어나야 하며, 이는 SDP 규율로 보장.
G₂ Graphify AST-based code graph

정의

소스 코드를 AST(Abstract Syntax Tree)로 파싱하여 함수/클래스/파일 노드와 호출/import/상속 엣지로 이루어진 그래프를 추출합니다. 언어 중립적 파서(tree-sitter 계열)를 사용해 다양한 언어를 지원하며, 추출 후 Leiden 커뮤니티 검출 알고리즘으로 코드를 자연스러운 모듈 단위로 클러스터링합니다.

이론적 근거 — 코드는 본질적으로 그래프

코드는 텍스트 파일의 모음이 아니라 호출 관계의 그래프입니다. LLM 컨텍스트에 코드를 prose로 제공하면 관계 정보가 손실되고, LLM은 이를 추론으로 복원하려다 자주 hallucination합니다. AST 파싱 결과를 그래프 형태로 유지하면 "무엇이 무엇을 호출하는가"가 결정적 사실이 됩니다 — LLM 추론 없이 파싱된 엣지를 그대로 반환하므로 hallucination 가능성이 0에 가깝습니다.

Leiden 커뮤니티 검출

Traag et al.(2019)이 Louvain 알고리즘의 단점(disconnected community 문제)을 개선해 제안한 방법. 코드 그래프에서 밀접하게 연결된 노드 집합을 자동으로 식별하며, 이는 보통 개발자가 인식하는 "모듈 경계"와 잘 일치합니다. 결과적으로 토큰 예산 내에서 "이 쿼리와 가장 관련 있는 코드 영역"을 효율적으로 선별할 수 있습니다.

담당 질문

  • "X 함수를 호출하는 함수 목록" (결정적)
  • "A에서 B까지의 호출 경로"
  • "이 엔티티의 구조적 이웃"
LIMITS "왜 이렇게 설계되었는가" 같은 의도/이력 질문에는 답할 수 없음 — Graphiti 담당. 동적 dispatch나 리플렉션은 AST로 완전히 추적 불가.
M MemoryMCP cross-session entity graph

정의

Anthropic의 Model Context Protocol (MCP) 위에 구축된 cross-session entity graph. 엔티티(사용자 선호, 프로젝트 규칙, 외부 시스템 포인터 등)를 name · type · observations · relations 구조로 저장하고, 모든 세션에서 동일한 인터페이스로 접근합니다.

이론적 근거 — MCP 표준의 의미

MCP는 LLM 에이전트가 외부 리소스(파일 시스템, 데이터베이스, 웹 등)에 표준화된 방식으로 접근하게 해주는 프로토콜입니다. Memory 서버를 MCP로 구현하면 에이전트 종류와 무관하게 동일한 지식 상태에 접근할 수 있습니다 — Claude Code, Codex CLI, Gemini CLI 등이 모두 같은 memory store를 공유합니다. 이는 조직 내 여러 에이전트가 공존하는 환경에서 필수적인 설계입니다.

담당 질문

  • "사용자가 이 상황에서 선호하는 방식"
  • "프로젝트 전체에 적용되는 규칙"
  • "외부 시스템의 참조 포인터 (예: 대시보드 URL, 이슈 트래커 프로젝트 ID)"
LIMITS 저장된 사실의 진위는 보장되지 않음 — 틀린 기억이 발견되면 NLF 규율에 따라 supersede 및 correction 필요.
§3 · 이론적 심화

knowledge graph와 wiki 증류의 이론적 위치

왜 이 두 가지 자료구조가 LLM 에이전트의 지식 관리에 핵심적인지, 학술적 배경과 함께 정리합니다.

knowledge graph 이론

관계형 데이터는 그래프 자료구조가 가장 자연스럽게 표현합니다. 벡터 임베딩은 유사도는 잘 잡지만, 인과 사슬(A → B → C)이나 비대칭 관계("A는 B에 의존하지만 역은 아님")를 구분하지 못합니다.

"의미가 비슷한 것"과 "실제로 연결된 것"은 다른 문제다.

knowledge graph는 이 두 문제를 분리해서 접근합니다. 엔티티 노드는 벡터 임베딩을 가지고(의미 유사도), 관계 엣지는 형식적 타입을 가집니다(관계의 종류). 검색은 두 신호를 모두 활용합니다.

Graphiti가 특히 hybrid search(vector + BM25 + 그래프 순회)를 채택한 이유입니다. 세 신호 중 하나라도 빠지면 특정 유형의 쿼리에서 recall이 급락합니다.

지식 증류 (Wiki 패턴)

Karpathy가 LLM 학습 데이터 품질을 논하며 제시한 관찰: Wikipedia가 좋은 학습 데이터인 이유는 내용 자체가 아니라 "정제되고 상호 링크된 canonical 형태"이기 때문이다.

Raw experience ≠ 지식. Distilled, cross-linked document = 지식.

이 관찰은 두 가지 설계 결정으로 이어집니다. 첫째, 경험은 atomic 단위로 기록 — 기록 장벽을 낮춰 compliance 확보. 둘째, N개가 쌓이면 LLM 합성으로 정제된 페이지 생성 — scale이 가능.

결과적으로 세션 n+1은 세션 n이 남긴 lesson뿐 아니라, 그 lesson이 포함된 distilled wiki에도 접근합니다. 지식이 복리처럼 누적되는 구조입니다.

Temporal resolution

코드베이스는 시간 의존적 사실로 가득합니다. "Phase 130에서 TM cascade는 single HNSW였다", "Phase 135에서 per-pair 구조로 변경". 두 사실은 모두 각자의 시점에 참이고, 단순히 전자를 삭제하면 과거 커밋 분석 시 맥락이 사라집니다.

Graphiti의 temporal layer는 사실에 valid_at / invalid_at을 부여해 "현재 참인 사실"과 "과거 참이었던 사실"을 동시에 유지합니다. 충돌 시 자동으로 최신이 이깁니다. NeuralTree도 유사하게 compile된 위키를 새 버전으로 supersede할 수 있습니다.

결정적 vs 확률적 분리

시스템에 LLM이 관여하는 곳과 안 하는 곳을 명확히 분리하는 것이 중요합니다. LLM 추론은 힘이 세지만 hallucination 위험이 있고, 확정성이 필요한 질문에는 부적합합니다.

Graphify는 일부러 LLM을 쓰지 않습니다 — AST는 컴파일러가 파싱한 결정적 결과이기 때문입니다. Graphiti는 LLM을 쓰되 추출 단계에만 한정하며, 쿼리 시점엔 저장된 그래프를 결정적으로 순회합니다. 이 분리가 깨지면 시스템 전체의 신뢰도가 떨어집니다.

§4 · 공학 규율

SDP와 NLF: 지식을 정직하게 활용하기 위한 규율

GNGM이 "무엇을 아는가"를 담당한다면, SDP는 "어떻게 변경을 수행하는가"를, NLF는 "무엇을 주장하지 않는가"를 담당합니다. 지식 스택만으로는 에이전트가 정직해지지 않습니다 — epistemic 규율이 함께 필요합니다.

SDP · Standard Development Protocol

모든 코드 변경이 통과해야 할 5단계

3줄짜리 버그 수정이라도 동일하게 적용. 각 단계의 목적은 서로 다른 오류 유형을 조기에 잡아내는 것.

  1. Brainstorm GNGM 4계층 병렬 조회 → 문맥 확보 → 계획 수립
  2. ECC Plan Review 4개 리뷰 에이전트가 코드 작성 전 계획 검토 (correctness · blast radius · pattern consistency · completeness)
  3. Execute 검토된 계획대로 구현. 계획 이탈 감지 시 중단하고 계획부터 갱신
  4. TDD Certificate 변경마다 RED → GREEN 증거 확보. 수동 확인은 증거로 인정되지 않음
  5. ECC Code Review 4개 전문 에이전트가 구현 검토 (SOLID · 단순성 · silent failure · 보안)

핵심 원리: 오류는 빨리 발견할수록 수정 비용이 낮습니다. 계획 단계 오류는 코드 수정 전에, 코드 오류는 머지 전에 잡습니다.

NLF · No Lie Fix

거짓 수정과 검증 없는 주장의 금지

사용자를 빨리 만족시키려는 조급함에서 발생하는 두 종류의 거짓말 — 땜빵 수정(fix bandage)과 확신 있는 미검증 주장(claim bandage) — 을 모두 금지합니다.

금지되는 행동

주석 처리 / 비활성화 / try-except로 삼키기 증상을 감추되 근본 원인은 그대로 남김
검증 없는 구조 주장 "X는 Y와 연결되어 있습니다" — 이번 세션에서 call chain을 trace하지 않은 채
grep 한 건으로 "없다" 단정 한 파일 검색 0건 ≠ 호출 체인 전체 검증 0건
주석·문서·핸드오프를 ground truth로 취급 코드와 주석이 다를 때 코드가 이김. 항상 코드로 재확인
검증된 주장의 정의 (4가지 질문 통과)
  1. 이번 세션에서 어떤 파일을 primary source로 읽었는가 (경로 + 라인 번호)
  2. entry point부터 동작 지점까지 call chain을 trace했는가
  3. 주석/문서가 아닌 실제 코드를 확인했는가
  4. grep 0건이라면 한 파일이 아닌 전체 체인을 검색했는가
§5 · 통합 워크플로우

실제 버그 수정 작업의 단계별 흐름

사용자가 버그 수정을 요청했을 때 GNGM + SDP + NLF가 어떻게 함께 작동하는지 예시. GNGM, NLF 같은 trigger phrase는 에이전트가 해당 규율을 명시적으로 활성화하는 키워드입니다.

# 1. Pre-task (GNGM 병렬 조회) graphiti.search("X 엔티티와 연결된 항목") # 관계·의사결정 이력 neuraltree.lesson_match([증상, 컴포넌트]) # 과거 동일 수정 유무 viking.search("X 관련 문서") # 의미 기반 문서 검색 memory.search_nodes("X 영역 사용자 선호") # cross-session 규칙 graphify.query("X를 호출하는 함수") # 결정적 구조 쿼리 # 2. SDP Step 1-2: 계획 수립 → 4 에이전트가 계획 검토 # correctness / blast radius / pattern consistency / completeness # CRITICAL / WARNING 항목 전부 해결 후 진행 # 3. NLF 체크 — 자가 진단 # "주석 처리하면 해결된다"는 생각이 드는가? → 근본 원인 미파악 # "X는 Y와 연결되어 있다"고 주장하려는가? → call chain 실제 trace 필요 # 4. SDP Step 3-5: 구현 → TDD → 4 에이전트 코드 리뷰 pytest -k "새로 추가된 테스트" # RED → GREEN 증거 # 5. Post-fix (GNGM 축적) neuraltree.lesson_add(domain, {symptom, root_cause, fix, chain}) graphiti.add_episode(name, body="... Connects: A → B → C ...") graphify.update() # post-commit hook이 자동 실행 memory.add_observations() # 새 영속 규칙이 발견되었을 경우에만 # 6. 증류 (조건 충족 시) # 해당 domain의 lesson이 3개 이상이 되었는가? neuraltree.compile(topic=domain) # LLM 합성으로 canonical wiki 생성 neuraltree.viking_index() # 의미 검색 인덱스에 재투입

이 워크플로우의 순효과: 세션 n+1이 세션 n의 모든 출력을 입력으로 받는다. 지식이 소실되지 않고 누적되며, 동일한 버그가 두 번 수동 분석되지 않습니다. 동시에 NLF가 검증 없는 주장을 차단하므로, 누적된 지식은 hallucination이 아닌 확인된 사실입니다.

§6 · 정리

세 가지 요소가 서로를 떠받치는 구조

GNGM은 "무엇을 아는가"의 문제를 해결합니다 — stateless LLM 에이전트에게 persistent, multi-representational 지식을 제공합니다.

SDP는 "어떻게 변경을 수행하는가"의 문제를 해결합니다 — 모든 코드 변경이 계획·검토·증거·재검토의 단계를 거치도록 강제합니다.

NLF는 "무엇을 주장하지 않는가"의 문제를 해결합니다 — 검증 없는 주장과 증상 덮기 식 수정을 epistemic 규율로 금지합니다.

셋 중 하나가 빠지면 다른 둘이 무력화됩니다. GNGM 없이 SDP/NLF만 있으면 매 세션이 맨손으로 시작합니다. SDP 없이 GNGM/NLF만 있으면 축적된 지식이 무분별하게 활용됩니다. NLF 없이 GNGM/SDP만 있으면 축적된 지식이 거짓 사실로 오염됩니다. 이 세 요소는 분리 가능한 모듈이 아니라 하나의 엔지니어링 시스템의 세 기둥입니다.

All components are local, open source, and language-agnostic.
